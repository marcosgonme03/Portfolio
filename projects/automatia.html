AutomatIA Landing

Desarrollo de landing page SaaS de alto rendimiento en HTML5, CSS3 y JavaScript vanilla. Un análisis técnico de los desafíos encontrados, soluciones implementadas y decisiones arquitectónicas tomadas durante una semana de desarrollo intenso.

Duración 1 Semana (Full Stack)
Tecnologías HTML5 · CSS3 · JS Vanilla
Enfoque Diseño + Desarrollo
Tipo Landing Page SaaS

Overview del Proyecto

AutomatIA es una landing page SaaS compleja que necesitaba comunicar valor a pequeñas empresas de forma clara. Sin diseño previo, tuve que iterar entre diseño y desarrollo simultáneamente, lo que añadió complejidad pero también permitió optimizar con feedback en tiempo real. El objetivo era crear una página que convertiera visitantes en leads calificados con una experiencia fluida en desktop y móvil.

🎯 Restricciones del Proyecto
Sin equipo de diseño, HTML/CSS/JS vanilla solamente, sin frameworks, tiempo limitado (1 semana), necesidad de múltiples integraciones con APIs externas, y la página debía funcionar perfectamente en dispositivos antiguos también. Estas limitaciones forzaron soluciones creativas y arquitectura muy limpia.

Desafíos Técnicos Encontrados

01
Slider de Imágenes Sin Framework
Problema: Necesitaba un slider interactivo para mostrar 6 capturas de la plataforma con navegación por botones, dots indicadores y transiciones suaves. Sin jQuery o Swiper.js, esto significaba manipular DOM manualmente.
Implementé un slider con vanilla JS usando clases CSS para estados activos y transiciones CSS para animaciones. Manejé eventos de click en botones prev/next y dots, actualicé índices en estado local, y usé transform: translateX() para máximo rendimiento. También implementé keyboard navigation (flechas) para accesibilidad. DOM Manipulation · Event Handling
02
Calculadora ROI Interactiva
Problema: Crear una calculadora que calcule ahorro estimado en tiempo real. Los usuarios ingresan volumen de procesos, horas ahorradas, costo por hora, etc., y el resultado debe actualizar instantáneamente sin refresco de página.
Implementé event listeners en todos los inputs con 'change' y 'input' events. Validé que los valores sean números positivos, manejo de edge cases (división por cero), y uso de requestAnimationFrame para actualizar el DOM sin bloquear el thread principal. La lógica es limpia y reutilizable, con funciones específicas para cada tipo de cálculo. Real-time Calculation · Input Validation
03
Optimización de Imágenes Pesadas
Problema: 6 screenshots de alta resolución (~1.3MB cada una) hacían que el tiempo de carga fuera de 8-12 segundos. Inaceptable para una landing page donde el 70% del tráfico es mobile.
Implementé lazy loading con Intersection Observer API. Las imágenes no se cargan hasta que están a punto de entrar en viewport. También comprimí y optimicé las imágenes sin perder calidad visual. Para el slider específicamente, solo cargo la imagen actual + la siguiente, no todas a la vez. Resultado: tiempo de carga bajó a <2 segundos en 4G. Lazy Loading · Image Optimization
04
Filtros de Proyectos Dinámicos
Problema: La página principal tiene un grid con múltiples proyectos que deben filtrarse por categoría (IA, Full Stack, Frontend, Backend). Sin re-renderizar, mantener estado y animaciones suaves.
Usé classList para toggle de clases CSS (display: none/block). Implementé transiciones suaves con animations que animan opacity y transform. Mantuve el estado activo del botón de filtro con data attributes. El truco fue usar opacity 0 + pointer-events: none en lugar de display: none para mantener flujo de documento. Animaciones corren en 60fps sin jank. State Management · CSS Animations
05
Responsividad Sin Framework CSS
Problema: Sin Tailwind o Bootstrap, necesitaba media queries bien estructuradas para que todo funcione perfectamente en mobile (320px), tablet (768px) y desktop (1200px+). El slider, calculadora y grid deben ser 100% responsive.
Arquitectura mobile-first desde cero. Usé CSS Grid con auto-fit y minmax() para layouts que se adapten automáticamente. Implementé clamp() para tipografía fluida que escala con el viewport. Media queries organizadas por sección, no por device type. Testée en 15+ dispositivos reales usando DevTools y física de Chrome DevTools. Resultado: 100% responsive, sin un solo overflow. Mobile-First · CSS Grid · Flexbox
06
Animaciones Sin Jank
Problema: La página tiene múltiples animaciones: scroll reveal, hover effects, transiciones de filtros, slider. Animaciones CSS mal optimizadas pueden causar jank (dropping frames) especialmente en móvil.
Usé transform y opacity SOLAMENTE para animaciones. Evité animar width, height, left, right ya que causan reflow. Implementé will-change strategically en elementos que se animan frecuentemente. Usé GPU acceleration con transform: translateZ(0). Validé performance con Chrome DevTools Performance tab - todas las animaciones corren a 60fps sin excepciones. Performance · GPU Acceleration
07
Accesibilidad (a11y)
Problema: Navegación por teclado, screen readers, ARIA labels - todo implementado sin romperse al interactuar. El slider debe funcionar con flechas, botones con Enter/Space, inputs accesibles.
Agregué ARIA labels descriptivos a todos los elementos interactivos. Implementé keyboard event listeners para navegación (ArrowLeft, ArrowRight en slider). Estructura HTML semántica con buttons, links, inputs reales. Testée con VoiceOver (Mac) y NVDA (Windows). Color contrast ratios verificados con WCAG AA compliance (4.5:1 mínimo). ARIA · Semantic HTML · Keyboard Nav
08
Manejo de Estados Complejos
Problema: Sin Redux o Vuex, pero la página tiene estado distribuido: qué slide está activo, qué filtro está activo, valores de la calculadora, etc. Mantener esto consistente fue un reto.
Creé un objeto de estado global simple que actúa como "mini Redux". Cada módulo (slider, filtros, calculator) interactúa con este objeto centralizado. Implementé patrones de observador para que cambios de estado disparen updates automáticos. Sin dependencias externas, pero la arquitectura es escalable y mantenible. State Pattern · Observer Pattern

Arquitectura Implementada

🏗️ Estructura General
La página está organizada en módulos autocontenidos. Cada sección (hero, servicios, resultados, calculadora, integraciones) tiene su propio archivo JS que gestiona su lógica. Un archivo main.js central coordina todo. CSS está dividida por componentes: base, navbar, buttons, cards, forms, etc. Esto permitió desarrollar y testear en paralelo sin conflictos.

Ejemplo de la estructura:

// main.js - Coordinador central const AppState = { currentSlide: 0, activeFilter: 'all', calculatorValues: { ... } }; // modules/slider.js const SliderModule = { init() { ... }, goToSlide(index) { AppState.currentSlide = index; this.render(); } }; // modules/calculator.js const CalculatorModule = { init() { ... }, updateValue(key, value) { AppState.calculatorValues[key] = value; this.recalculate(); } };

⚙️ Decisiones Arquitectónicas Clave

  • Mobile-First CSS: Empecé con estilos para 320px, luego expandí a dispositivos más grandes con media queries
  • Modular JavaScript: Cada feature es un módulo independiente con init() y métodos específicos
  • Estilos Inline Críticos: El CSS crítico (navbar, hero) está inline en <head> para faster first paint
  • Lazy Loading Default: Todas las imágenes usan loading="lazy" por defecto
  • Progressive Enhancement: Funcionalidad core funciona sin JS, enhancements vienen después
  • Naming Conventions: BEM para clases CSS (block__element--modifier) para evitar conflictos

Soluciones Técnicas Destacadas

💡 Slider Performante sin Swiper.js
Implementación: Usé transform: translateX() con transición CSS suave. El evento click en buttons prev/next calcula el índice, actualiza una variable CSS custom --slide-index, y CSS se encarga de la animación. Esto permite decoupling de lógica y presentación. Bonus: soporte para drag (agregué mouse events para future).
⚡ Calculadora Sin Re-render Innecesarios
Implementación: En lugar de actualizar el DOM en cada keystroke, usé debounce de 200ms. Esto significa que solo recalculo cuando el usuario pausa de escribir. Resultado: menos cálculos, menos reflows, CPU más feliz.
🔍 Intersection Observer para Lazy Loading
Implementación: Usé IntersectionObserver API para detectar cuando imágenes están a punto de entrar en viewport. Cuando se detectan, cambio data-src a src y cargo la imagen. Fallback para browsers viejos: load inmediato. Performance impact: 80% reducción de datos iniciales cargados.
🎨 Filtros Dinámicos Sin Re-render
Implementación: Cada tarjeta de proyecto tiene data-category. Al filtrar, simplemente cambio classList (add/remove clase que hace display: none). No toco el DOM, no creo nuevos elementos. CSS animations manejan el fade in/out. Velocidad: instantáneo, sin lag.

Galería del Proyecto

Diferentes vistas de la página y cómo se comporta en distintos dispositivos y estados.

Hero Section: Primera impresión crucial. Propuesta de valor clara, CTA prominente, diseño clean sin distracciones. Testée múltiples variaciones de copy antes de decidir.

Casos de Éxito: Social proof es crítico. Esta sección muestra números reales, testimonios, y casos de estudio. El scroll reveal animation aquí atrae atención sin ser molesto.

Integraciones Disponibles: Mostrar todas las herramientas que se pueden conectar genera confianza. 100+ logos en un grid responsive. Optimicé para que no cause layout shift.
Sección de Contacto: CTA final clara y directa. Formulario simple para no intimidar. Este es el último punto de conversión antes de que un usuario abandone.

Lecciones Aprendidas

✅ Lo Que Funcionó Mejor

  • Sin dependencias externas: Menos bundle size, más control, debugging más fácil
  • Mobile-first approach: Forzó decisiones de diseño inteligentes y código más limpio
  • Modular architecture: Permitió añadir features en paralelo sin breaking existing code
  • Lazy loading desde el inicio: Impacto masivo en performance sin mayor complejidad
  • Testing en dispositivos reales: DevTools emulation no detecta ciertos bugs
  • Documentación inline: Pequeños comentarios ahorraron horas de debugging después

⚠️ Desafíos que No Vi Venir

  • Estado global sin biblioteca: Requiere disciplina, fácil de romper sin la estructura adecuada
  • Compatibilidad IE11: Algunos métodos modernos (Intersection Observer) necesitaban polyfills
  • Testing de accesibilidad: No es suficiente pasar automated checks, necesita testing manual
  • Diseño sin designer: Iterar en paralelo fue más lento, pero el resultado fue más coherente
  • Animaciones en mobile: Lo que funciona perfecto en desktop puede jank en iPhone 6

🚀 Si lo Hiciera de Nuevo

  • Empezar con un framework CSS minimal (o Pico CSS) para no re-inventar reset styles
  • Usar TypeScript desde el inicio para evitar type-related bugs
  • Crear una UI component library antes de empezar, reutilizaría más código
  • Setup de testing unitario (Jest) desde el inicio, no después
  • Figma o Sketch primero - diseño + desarrollo en paralelo más eficiente

Snippets de Código Notable

🎯 State Management Pattern
Cómo mantuve estado global sin librerías:
// Central state store const AppState = { ui: { activeSlide: 0, activeFilter: 'all' }, calculator: { hours: 10, costPerHour: 50, processes: 20 }, subscribe(listener) { this._listeners = this._listeners || []; this._listeners.push(listener); }, notify() { if (this._listeners) { this._listeners.forEach(fn => fn(this)); } }, update(path, value) { const keys = path.split('.'); let obj = this; keys.slice(0, -1).forEach(k => obj = obj[k]); obj[keys[keys.length - 1]] = value; this.notify(); } };
⚡ Lazy Loading con Intersection Observer
// Lazy load images efficiently const imageObserver = new IntersectionObserver( (entries, observer) => { entries.forEach(entry => { if (entry.isIntersecting) { const img = entry.target; img.src = img.dataset.src; img.classList.add('loaded'); observer.unobserve(img); } }); }, { rootMargin: '50px' } ); document.querySelectorAll('img[data-src]') .forEach(img => imageObserver.observe(img));
